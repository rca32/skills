# `code-review`: 두 기준으로 독립적인 코드 리뷰

정확한 실행 계약: [`skills/code-review/SKILL.md`](../../skills/code-review/SKILL.md)

## 한마디로

브랜치, PR, commit 범위 또는 현재 작업 트리의 변경 전체를 고정한 뒤 **저장소 규칙(Standards)**과 **원래 요구사항(Spec)**이라는 두 기준으로 독립적으로 검토하는 읽기 전용 스킬입니다.

규칙을 잘 지킨 코드가 잘못된 기능일 수 있고, 요구사항을 만족하는 코드가 저장소의 안전·설계 규칙을 어길 수도 있습니다. 그래서 두 검토를 섞지 않습니다.

## 무엇을 검토하나요?

검토 범위는 commit에 들어간 내용만이 아닙니다.

- 기준 commit부터 `HEAD`까지의 commit
- staged 변경
- unstaged 변경
- Git이 아직 추적하지 않는 파일

먼저 비교 기준의 SHA, 현재 `HEAD`, 작업 트리 상태와 diff를 기록합니다. 시작과 보고 직전에 diff와 untracked 파일의 fingerprint를 비교해 검토 도중 내용이 바뀌지 않았는지도 확인합니다.

기준점이 모호하면 임의로 고르지 않습니다. 사용자가 지정한 ref를 우선하고, 없으면 하나로 확정되는 upstream 또는 원격 기본 브랜치를 사용합니다. 후보가 충돌하면 기준점을 물어봅니다.

## 두 검토 기준

### Standards

변경된 경로에 적용되는 `AGENTS.md`, 기여 지침, 코딩 규칙, 아키텍처 원칙을 기준으로 봅니다. 일반적인 code smell 목록은 보조 힌트일 뿐이며 저장소 규칙이 항상 우선합니다.

### Spec

사용자가 준 이슈, PRD, 인수 조건 또는 명세를 기준으로 봅니다. 구현 코드를 보고 요구사항을 거꾸로 추측하지 않습니다. 권위 있는 명세를 찾지 못하면 합격으로 처리하지 않고 `Not evaluated`라고 표시합니다.

가능하면 두 개의 깨끗한 reviewer context가 같은 고정 snapshot을 서로의 결론을 보지 않고 각각 검토합니다. 독립 검토가 불가능하면 한 사람이 두 기준을 봤다고 가장하지 않고 그 제한을 보고합니다.

## 어떤 내용만 finding이 되나요?

다음 조건을 모두 만족하는 문제만 finding으로 보고합니다.

- 이번 변경에서 새로 생겼거나 악화됨
- 실제 변경 줄과 실행·의존 경로를 확인함
- 다른 구현, 테스트 또는 예외 규칙이 주장을 무효화하지 않음
- 적용되는 규칙 또는 요구사항이 결론을 뒷받침함
- 영향이 구체적이고 수정 방향이 실행 가능함

기존부터 있던 문제, 취향 차이, 막연한 미래 위험이나 도구로 바로 확인할 lint 추측은 finding으로 세지 않습니다. 심각도는 `blocker`, `high`, `medium`, `low`를 사용하며 두 축의 결과를 합쳐 하나의 종합 점수로 바꾸지 않습니다.

## 읽기 전용이라는 의미

이 스킬은 파일 수정, 이슈·PR 변경, lease 획득, commit 또는 push를 하지 않습니다. 영구적인 리뷰 보고서가 필요하다면 `documenting-work`로 권장 위치를 정할 수 있지만, 실제 파일 쓰기는 권한 있는 바깥 작업 흐름이 담당합니다.

## 요청 예시

```text
$code-review main과의 merge-base 이후 현재 작업 전체를 이슈 #42의 요구사항과 저장소 규칙 기준으로 검토해 줘.
```
