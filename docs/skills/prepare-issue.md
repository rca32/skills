# `prepare-issue`: 개발 이슈 준비하기

정확한 실행 계약: [`skills/prepare-issue/SKILL.md`](../../skills/prepare-issue/SKILL.md)

## 한마디로

새로 들어온 GitHub 이슈나 외부 Pull Request를 조사해 **무슨 종류의 요청인지**, **주장이 사실인지**, **에이전트가 바로 작업해도 되는지** 판단하는 스킬입니다.

`prepare-issue`는 구현을 시작하는 스킬이 아닙니다. 작업을 시작하기 전에 요청을 믿을 만하고 실행 가능한 형태로 만드는 접수 창구에 가깝습니다.

## 언제 사용하나요?

- 이슈 설명이 모호하거나 정보가 부족할 때
- 버그 신고가 실제로 재현되는지 확인할 때
- 기능 요청이 현재 동작과 어떤 차이가 있는지 확인할 때
- 중복 이슈나 이미 구현된 요청인지 찾을 때
- 이슈를 `에이전트 작업 가능`, `정보 필요`, `진행하지 않음` 같은 상태로 정리할 때
- 외부 기여 PR이 주장한 변경을 실제로 포함하는지 검토할 때

이미 준비가 끝난 이슈를 맡아 구현하려는 경우에는 `work-github-issue`를 사용합니다.

## 실제로 무엇을 하나요?

먼저 저장소가 사용하는 이슈 라벨과 상태 규칙을 읽습니다. 기본 규칙에서는 요청 종류인 `유형: 버그` 또는 `유형: 개선` 하나와, 다음 진행 상태 하나를 분리해 관리합니다.

| 표시 라벨 | 뜻 |
| --- | --- |
| `상태: 분류 필요` | 사실 확인이나 범위 결정이 더 필요함 |
| `상태: 정보 필요` | 사람이 구체적인 질문에 답해야 함 |
| `상태: 에이전트 작업 가능` | 설명과 선행 작업이 완전해 구현을 시작할 수 있음 |
| `상태: 사람 검토 필요` | 승인, 권한 부여, 병합 또는 수동 작업을 사람이 해야 함 |
| `상태: 진행하지 않음` | 중복, 기구현 또는 거절로 진행하지 않음 |

`ready-for-agent`, `bug`, `enhancement` 같은 영문 이름은 자동화의 내부 역할 키와 기존 이슈 호환용 별칭으로만 사용합니다. 기존 한국어 라벨이 있으면 사용하고, 없으면 이슈 body의 상태 marker로 진행합니다. 새 이슈에는 같은 의미의 한글·영문 라벨을 동시에 붙이지 않습니다.

그다음 이슈 본문, 댓글, 라벨, 관계, 작성자와 변경 시점을 확인합니다. 저장소 안에서는 다음 증거를 넓게 검색합니다.

- 이미 존재하는 구현
- 같은 문제를 다루는 중복 또는 후속 이슈
- 요청을 제한하거나 거절한 기존 결정
- 이미 진행 중인 관련 작업

버그라면 안전한 환경에서 재현하거나 최소한의 반복 가능한 관찰을 만듭니다. 기능 요청이라면 현재 동작과 요청한 동작 사이의 차이를 확인합니다. 결과는 `confirmed`, `not reproduced`, `insufficient information` 중 하나로 분류합니다.

마지막으로 에이전트가 작업할 수 있는 구체적인 brief를 만들거나, 부족한 정보만 묻거나, 중복·기구현·거절 이유를 기록합니다.

## 사람이 개입해야 한다면

`상태: 정보 필요`나 `상태: 사람 검토 필요`를 붙이는 것만으로는 부족합니다. 이슈 본문이나 최신 댓글에 다음 내용을 함께 남깁니다.

- 에이전트가 멈춘 이유
- 요청 종류와 정확한 대상
- 사람이 답할 질문이나 수행할 작업 하나
- 답변·승인·결과를 남길 위치
- 완료됐다고 판단할 수 있는 증거
- 그 증거를 찾을 수 있는 댓글·리뷰·로그·링크·ID
- 완료 후 `상태: 에이전트 작업 가능`으로 바꿀지, 추가 정보를 받을지, 이슈를 종료할지
- 상태 전환을 담당할 `prepare-issue` 또는 완료를 담당할 `work-github-issue`
- `상태: 사람 검토 필요`라면 사람이 실제 결과로 수정해 바로 남길 수 있는 추천 댓글

추천 댓글은 정확한 대상, 승인·수정 요청 같은 허용된 결과 선택지, 판단 근거와 증거 링크 자리를 포함합니다. 실제 검토·승인·수동 작업을 대신하지 않으며, 사람은 작업 후 실제 결과로 수정해 안내된 위치에 게시합니다. 사람은 상태 라벨을 직접 바꾸지 않습니다. 답변이 등록됐다고 자동으로 작업 가능 상태가 되는 것도 아닙니다. GitHub 변경 권한을 받은 `prepare-issue`가 새 증거를 반영해 brief와 blocker를 다시 확인한 뒤 상태를 전환합니다.

## 상태를 함부로 바꾸지 않는 이유

사용자가 평가만 요청했다면 추천 상태와 근거만 보고하고 GitHub는 수정하지 않습니다. 라벨, 댓글, 종료 같은 변경은 명시적인 허가가 있어야 합니다.

허가받아 GitHub를 변경할 때도 `work-github-issue`의 짧은 planning lease로 같은 이슈를 동시에 정리하는 세션이 없도록 막습니다. 사용자가 `상태: 에이전트 작업 가능`을 요청했다는 이유만으로 준비되지 않은 이슈를 준비 완료로 표시하지 않습니다.

## 하지 않는 일

- 구현 lease 획득
- 제품 코드 수정
- 커밋, push 또는 구현 증거 게시
- 준비된 이슈의 실제 구현

## 요청 예시

```text
$prepare-issue 이슈 #42가 실제 버그인지 확인하고, 작업 가능한 상태인지 평가해 줘.
```

평가뿐 아니라 라벨과 brief도 실제로 반영하려면 그 변경까지 명시해서 요청해야 합니다.
