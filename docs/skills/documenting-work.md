# `documenting-work`: 개발 문서의 자리 정하기

정확한 실행 계약: [`skills/documenting-work/SKILL.md`](../../skills/documenting-work/SKILL.md)

## 한마디로

명세, 결정, 조사, 진단, 리뷰 같은 개발 지식을 **어디에 얼마나 오래 보관하고 무엇을 원본으로 삼을지** 결정하는 스킬입니다.

같은 내용을 GitHub, Markdown 파일과 실행 결과에 여러 벌로 복사하면 곧 서로 달라집니다. 이 스킬은 권위 있는 원본을 정확히 하나만 두고 다른 위치에는 링크만 남기게 합니다.

## 먼저 보관 등급을 고릅니다

| 등급 | 적합한 내용 | 기본 처리 |
| --- | --- | --- |
| 대화 | 일회성 분석, 초안, 진단, 리뷰 | 파일을 만들지 않고 응답으로 반환 |
| Tracker | 이슈 brief, 티켓 그래프, 구현 증거, 인계 | GitHub 이슈·PR·댓글·관계가 원본 |
| 저장소 | 코드와 함께 검토·버전 관리할 명세나 결정 | 정해진 `docs/` 규칙에 따라 파일 저장 |
| Artifact | 로그, trace, screenshot, benchmark | 프로젝트의 artifact 저장·보존 정책 사용 |

“설명해 줘”, “검토해 줘”, “초안을 만들어 줘”는 기본적으로 대화 등급입니다. “저장해 줘”, “기록해 줘”, “이 파일에 작성해 줘”처럼 지속성을 요청해야 저장소 문서를 만듭니다.

## 문서 위치는 어떻게 정하나요?

다음 우선순위로 규칙을 찾습니다.

1. 현재 저장소의 안전·소유권·문서 지침
2. 지침과 충돌하지 않는 사용자의 명시적인 목적지
3. 같은 종류의 기존 문서와 인덱스 관례
4. 이 스킬에 포함된 기본 fallback 규칙

서로 충돌하는 활성 관례가 두 개라면 임의로 하나를 택하지 않습니다. 어느 위치가 원본인지 결정될 때까지 대화 안에만 결과를 둡니다.

## 저장할 때 무엇을 확인하나요?

- 같은 문서 ID나 제목, 원본 이슈가 이미 존재하는지 검색
- 하나의 권위 있는 본문만 생성 또는 갱신
- 저장소 문서라면 가장 가까운 문서 인덱스도 갱신
- 문서의 상태를 `draft`, `active`, `superseded`, `archived` 등으로 관리
- 새 문서가 이전 문서를 대체하면 양쪽에 서로의 링크를 남김
- 파일과 인덱스를 다시 읽어 실제 반영 결과 확인

이슈에 연결된 공유 문서를 변경하는 경우에는 `work-github-issue`의 planning 또는 implementation lease도 확인합니다.

## 기본 저장 위치

저장소에 별도 문서 규칙이 없을 때만 다음 fallback을 사용합니다.

```text
docs/README.md
docs/specs/
docs/decisions/
docs/research/
docs/reports/diagnostics/
docs/reports/reviews/
```

원시 로그나 screenshot을 편의상 `docs/`에 넣지는 않습니다. 그런 결과는 artifact 정책을 따릅니다.

## 하지 않는 일

- 문서의 실제 도메인 내용 결정
- GitHub 이슈 생명주기나 lease 프로토콜 소유
- 같은 본문을 여러 시스템에 중복 유지
- 명시적 허가 없이 기존 문서 삭제

## 요청 예시

```text
$documenting-work 이 설계 결정은 어디를 원본으로 삼아 어떤 이름으로 저장해야 하는지 정해 줘.
```
