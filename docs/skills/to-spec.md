# `to-spec`: 대화를 명세로 바꾸기

정확한 실행 계약: [`skills/to-spec/SKILL.md`](../../skills/to-spec/SKILL.md)

## 한마디로

현재 대화와 저장소에서 **이미 합의된 결정**을 모아 제품 또는 개발 명세로 정리하는 스킬입니다.

빈칸을 그럴듯한 요구사항으로 채우지 않습니다. 결정되지 않은 내용은 가정이나 열린 질문으로 분리합니다.

## 언제 사용하나요?

- 긴 대화에서 합의한 내용을 하나의 명세로 정리할 때
- 구현 전에 현재 동작과 원하는 동작을 명확히 고정할 때
- 큰 작업을 `to-tickets`로 나누기 전에 승인 가능한 원본을 만들 때
- 요구사항, 제외 범위, 테스트 관찰 지점을 함께 기록할 때

아직 요구사항을 탐색하는 인터뷰가 필요한 상황보다는, 논의가 어느 정도 끝난 뒤 정리할 때 적합합니다.

## 실제로 무엇을 하나요?

대화에서 문제, 원하는 결과, 제약, 안전 경계, 제외 범위와 미결정을 추출합니다. 이어서 저장소의 도메인 용어, 기존 아키텍처 결정과 공개 인터페이스를 읽어 현재 코드와 충돌하지 않는지 확인합니다.

명세에는 보통 다음 내용이 들어갑니다.

- 현재 동작과 해결하려는 문제
- 원하는 사용자 또는 시스템 동작
- 확정된 요구사항과 측정 가능한 인수 조건
- 하지 않기로 한 범위
- 테스트에서 동작을 관찰할 공개 seam
- 명시적인 가정과 열린 질문

모든 인수 조건은 실제 요구사항으로 추적되어야 하고, 중요한 제약에는 확인 방법이 있어야 합니다.

## 열린 질문이 있으면 어떻게 하나요?

저장소 안전 규칙과 사용자 결정이 충돌하거나, 선택에 따라 구현 범위가 크게 달라지는 내용은 임의로 결정하지 않습니다. `Open questions`에 남기고 명세를 draft 또는 질문 해결 대기 상태로 둡니다.

이 흐름에서는 새로운 요구사항 인터뷰를 다시 시작하지 않습니다. 확인되지 않은 내용을 명세 속 확정 사항처럼 쓰지 않는 것이 핵심입니다.

## 어디에 저장하나요?

초안만 요청했다면 대화로 반환하며 파일이나 GitHub 이슈를 만들지 않습니다. 저장이 요청되면 `documenting-work`가 저장소 문서와 GitHub 중 권위 있는 원본 하나를 정합니다.

게시된 명세는 구현 티켓이 아니라 계획 원본입니다. 따라서 곧바로 `ready-for-agent`를 붙이지 않고, 구현 이슈 분해는 `to-tickets`에 맡깁니다.

## 하지 않는 일

- 확인되지 않은 요구사항 발명
- 구현 이슈 claim
- 명세를 구현 티켓으로 위장
- 같은 명세 본문을 GitHub와 Markdown 양쪽에 중복 저장

## 요청 예시

```text
$to-spec 지금까지 합의한 결제 재시도 정책을 구현 전 검토할 명세로 정리해 줘.
```
