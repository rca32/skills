# `tdd`: 작은 Red-Green-Refactor 반복

정확한 실행 계약: [`skills/tdd/SKILL.md`](../../skills/tdd/SKILL.md)

## 한마디로

이미 원하는 동작이나 버그 원인이 정해진 작업을 **실패하는 테스트부터 시작해 작은 단위로 구현**하는 스킬입니다.

테스트를 한꺼번에 많이 작성한 뒤 구현을 몰아서 하는 방식이 아니라, 관찰 가능한 동작 하나마다 Red, Green, Refactor를 끝내고 다음 동작으로 이동합니다.

## 언제 사용하나요?

- 요구사항이 분명한 기능을 테스트 우선으로 구현할 때
- 원인이 확인된 버그에 회귀 테스트를 추가하고 수정할 때
- integration test나 공개 인터페이스 테스트부터 변경을 만들 때

원인을 아직 모르는 실패라면 먼저 `diagnosing-bugs`를 사용합니다.

## 공개 seam부터 정합니다

seam은 외부 호출자가 동작을 관찰하는 안정적인 접점입니다. 예를 들면 다음과 같습니다.

- 공개 함수
- CLI 명령
- HTTP endpoint
- event
- 화면 상호작용

테스트는 private method의 호출 순서나 내부 구현이 아니라 이 공개 seam을 통해 동작을 확인합니다. 그래야 내부 구조를 정리해도 테스트가 불필요하게 깨지지 않습니다.

## 한 번의 반복

### Red

외부에서 관찰 가능한 동작 하나를 표현하는 가장 작은 테스트를 추가합니다. 가장 좁은 테스트 명령을 실행하고, 문법이나 fixture 문제가 아니라 아직 구현되지 않은 바로 그 이유로 실패하는지 확인합니다.

### Green

현재 실패를 통과시키는 가장 작은 제품 코드 변경만 적용합니다. 다음 단계의 기능을 미리 구현하거나 공개 계약을 근거 없이 넓히지 않습니다.

### Refactor

완료된 동작에서 드러난 이름, 중복 또는 구조 문제만 정리합니다. 이 단계에서 새 동작을 추가하지 않으며 테스트가 계속 통과하는지 확인합니다.

이 한 주기가 완전히 green이고 일관된 상태가 된 뒤 다음 동작으로 넘어갑니다.

## 좋은 테스트의 기준

- 구현이 아니라 안정적인 외부 행동을 검증
- 제품 코드와 같은 알고리즘으로 기대값을 다시 계산하지 않음
- 시간, 난수와 변경 가능한 상태를 고정해 결정적으로 실패
- 외부 시스템은 필요한 만큼 격리하고 내부 협력 객체는 가능한 실제 구현 사용
- 저장소의 기존 테스트 도구와 fixture 관례를 우선 사용

실제 결제, 주문, 메시지나 production traffic은 일반 테스트 허가만으로 실행하지 않습니다.

## 이 스킬이 소유하지 않는 일

`tdd`는 구현 안쪽의 테스트-코드 반복만 담당합니다. GitHub 이슈 claim이나 lease, tracker 상태, commit, push, PR 게시와 완료 증거는 바깥의 `work-github-issue` 또는 사용자가 허가한 다른 흐름이 담당합니다.

## 요청 예시

```text
$tdd 비어 있는 검색어를 거부하는 동작을 공개 API 테스트부터 구현해 줘.
```
