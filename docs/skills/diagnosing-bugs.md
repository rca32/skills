# `diagnosing-bugs`: 증거로 버그 원인 찾기

정확한 실행 계약: [`skills/diagnosing-bugs/SKILL.md`](../../skills/diagnosing-bugs/SKILL.md)

## 한마디로

원인을 모르는 오류, 간헐적 실패 또는 성능 저하를 **빠르게 반복 가능한 신호와 반증 실험**으로 좁혀 실제 원인을 찾는 스킬입니다.

로그에서 그럴듯한 이야기를 고르는 것이 아니라, 경쟁하는 원인 후보들을 하나씩 틀렸다고 증명하면서 남은 원인을 확정합니다.

## 먼저 요청 범위를 구분합니다

- **진단만:** 재현하고 원인을 설명하지만 제품 코드는 고치지 않음
- **진단 후 수정:** 회귀 테스트, 최소 수정, 검증과 임시 장치 정리까지 수행

요청이 애매하면 안전하게 진단만으로 해석합니다. 이슈에 연결된 파일을 수정하려면 바깥의 `work-github-issue` 구현 lease가 먼저 있어야 합니다.

## 실제로 무엇을 하나요?

### 1. 증상을 정확히 정의

기대 동작, 실제 동작, 가장 작은 알려진 조건, 환경과 시작 시점을 정리합니다. 성능 문제라면 측정할 지표, workload, 기준값과 허용 한계를 먼저 정합니다.

### 2. 빠른 피드백 루프 생성

실제 동작 경로를 실행하면서 문제와 다른 실패를 구분하는 명령 하나를 만듭니다. 기존 focused test가 가장 좋고, 없으면 공개 seam의 작은 harness, CLI·HTTP replay, headless UI, seed가 고정된 반복 실험 등을 사용합니다.

간헐적 실패는 여러 번 실행한 횟수와 실패율을 기록하고, 성능 문제는 한 번의 시간 대신 반복 측정을 사용합니다.

### 3. 재현 조건 최소화

입력, 설정, 호출자, 데이터와 절차를 하나씩 제거하며 매번 다시 실행합니다. 남은 요소를 하나라도 제거하면 문제가 사라지는 상태가 최소 재현입니다. 원래 시나리오도 보존해 나중에 함께 검증합니다.

### 4. 가설을 반증

가능한 원인 세 개에서 다섯 개를 근거 순서로 정리하고 각 가설에 예측을 붙입니다.

```text
X가 원인이라면 Y를 바꾸거나 관찰했을 때 Z가 나타나야 한다.
```

한 번에 변수 하나만 바꾸며 실험합니다. root cause는 단순히 증상과 어울리는 설명이 아니라, 다른 주요 가설과 구별되는 증거가 있을 때 확정됩니다.

### 5. 보고하거나 수정

진단만 요청받았다면 재현 명령, 최소 사례, 확정 원인, 탈락한 가설의 증거, 영향 범위, 남은 불확실성, 추천 수정과 회귀 테스트 위치를 보고합니다.

수정도 허가받았다면 최소 재현을 공개 seam의 실패하는 회귀 테스트로 만들고 `tdd`로 가장 작은 수정을 적용합니다. 회귀 테스트뿐 아니라 원래 시나리오와 관련 테스트도 다시 실행합니다.

## 안전 경계

실제 주문, 결제, 메시지, 파괴적 쓰기 또는 production traffic을 단순 재현 목적으로 실행하지 않습니다. 그런 경로는 별도의 명시적 허가와 rollback 또는 reconciliation 장치가 있을 때만 사용합니다. 그렇지 않으면 sandbox, fixture, replay 또는 fake로 대체합니다.

사용자의 현재 작업 트리에서 revision을 바꾸며 bisect하지 않습니다. 필요한 경우 disposable worktree에서만 수행합니다.

## 요청 예시

```text
$diagnosing-bugs 간헐적인 timeout의 원인만 찾아서 증거와 함께 설명해 줘. 아직 수정하지 마.
```
