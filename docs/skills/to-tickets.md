# `to-tickets`: 명세를 티켓으로 나누기

정확한 실행 계약: [`skills/to-tickets/SKILL.md`](../../skills/to-tickets/SKILL.md)

## 한마디로

승인된 명세나 계획을 **작고 검증 가능하며 의존성이 안전한 구현 티켓**으로 분해하는 스킬입니다.

단순히 작업 목록을 잘게 나누는 것이 아니라, 각 티켓이 한 세션 안에서 완결된 결과를 내고 전체 요구사항이 빠짐없이 연결되게 만듭니다.

## 좋은 티켓은 어떤 모습인가요?

각 티켓은 다음 조건을 만족해야 합니다.

- 필요한 계층을 통과하는 좁은 end-to-end 동작을 제공함
- 독립적으로 시연하거나 검증할 수 있음
- 새 에이전트 세션 하나에서 처리할 수 있음
- 구체적인 인수 조건과 제외 범위가 있음
- 실제 착수를 막는 작업만 blocker로 가짐

한 기능을 DB, 백엔드, 프론트엔드, 테스트로 기계적으로 나누지 않습니다. 가능한 경우 사용자가 확인할 수 있는 동작과 그 테스트를 같은 티켓에 넣는 vertical tracer bullet 방식을 사용합니다.

## 실제로 무엇을 하나요?

승인된 명세와 관련 결정 기록을 모두 읽고, 미해결 질문이 티켓 경계를 바꾸지 않는지 확인합니다. 질문이 범위나 의존성을 바꾼다면 `to-spec`으로 돌아갑니다.

이슈 제목과 본문은 저장소 또는 사용자가 정한 언어로 작성하고, 별도 지정이 없으면 한국어를 사용합니다. 제목, 완료 조건, 제외 범위와 안내 문구는 이해하기 쉬운 한국어로 쓰되 API 이름, 코드 식별자, 링크와 자동화 marker는 원문 그대로 유지합니다.

초안 티켓에는 `T1`, `T2` 같은 안정적인 키를 붙이고 그 키로 blocker를 표현합니다. 명세와 사용한 결정 기록에서는 SHA-256 revision fingerprint를 만들어 어떤 원본 버전에서 분해했는지 추적합니다.

게시 전에 다음을 검사합니다.

- 존재하지 않는 blocker나 순환 의존성이 없는가
- 모든 의존성이 진짜 착수 차단 조건인가
- 모든 원본 요구사항이 인수 조건에 연결되는가
- 각 티켓에 관찰 가능한 결과가 있는가
- 승인 범위를 몰래 확장하지 않았는가
- 처음 시작할 수 있는 티켓에 숨은 선행 조건이 없는가

병렬로 시작할 수 있는 initial frontier와 전체 일정을 좌우하는 critical path도 함께 보여 줍니다.

## 왜 두 단계로 게시하나요?

사용자가 GitHub 게시를 명시적으로 허가하면 먼저 모든 티켓을 `상태: 분류 필요`로 생성하고 부모·자식 관계와 blocker를 연결합니다. 이 단계에서는 `상태: 에이전트 작업 가능`을 붙이지 않습니다.

게시 전에 선택된 tracker 계약의 라벨이 저장소에 실제로 존재하는지도 확인합니다. 별도 계약이 없을 때는 기본 한국어 라벨을 사용합니다. 이슈 게시 요청만으로 저장소 전체 라벨을 새로 만들 권한까지 생기지는 않으므로, 라벨이 없으면 필요한 이름과 의미를 알려 주고 별도 설정 허가를 기다립니다.

전체 게시 그래프를 다시 읽어 관계와 본문을 검증한 뒤에만 실제로 시작 가능한 구현 티켓을 `상태: 에이전트 작업 가능`으로 바꿉니다. 연결이 덜 된 티켓을 다른 에이전트가 먼저 가져가는 사고를 막기 위한 절차입니다.

사람의 답변이나 승인이 먼저 필요한 티켓은 작업 가능 상태로 바꾸지 않습니다. 대신 `상태: 정보 필요` 또는 `상태: 사람 검토 필요`를 붙이고 본문에 다음 내용을 쉽게 적습니다.

- 왜 사람의 도움이 필요한지
- 요청 종류와 정확한 대상
- 사람이 정확히 무엇을 해야 하는지
- 답변이나 결과를 어디에 남겨야 하는지
- 무엇이 확인되면 끝난 것인지
- 완료를 입증할 댓글·리뷰·로그·링크·ID가 무엇인지
- 그다음 어떤 상태로 바뀌는지
- 그 상태를 검증하고 적용할 스킬이 무엇인지
- `상태: 사람 검토 필요`라면 정확한 대상과 실제 허용 결과, 판단 근거, 증거 링크 자리를 갖춘 추천 댓글

추천 댓글은 예를 들어 `PR #128 권한 변경 — 결과: [승인 | 수정 요청]. 판단 근거: [작성]. 완료 증거: [PR 리뷰 링크 붙여넣기].`처럼 사람이 실제 작업 후 수정해 바로 게시할 수 있어야 합니다. 따라서 “검토해 주세요”처럼 다음 행동을 추측해야 하는 안내는 완성된 티켓으로 보지 않습니다.

각 이슈에는 원본, revision, 초안 키를 담은 숨은 marker를 넣습니다. 생성 응답이 끊겨 결과를 알 수 없을 때 marker 검색과 전체 이슈 목록을 모두 확인해 기존 이슈를 찾습니다. 아무것도 검색되지 않았다는 사실만으로 생성 실패라고 판단하지 않습니다. 원래 요청이 생성 전에 거절됐다는 지속적인 응답이나 요청 기록이 확인될 때만 다시 생성하고, 그렇지 않으면 결과를 unknown으로 유지합니다. 그동안 만든 티켓은 준비 중 상태로 두고 안전한 복구 방법을 보고합니다.

## 권한과 책임 경계

초안이나 검토만 요청하면 GitHub를 수정하지 않습니다. 게시가 명시적으로 요청된 경우에만 `work-github-issue`의 planning lease 아래에서 이슈, 관계와 준비 상태를 변경합니다.

이 스킬은 티켓을 claim하거나 구현하지 않으며, 코드 커밋, 구현 증거, 부모 이슈 종료도 담당하지 않습니다. 그 이후의 생명주기는 `work-github-issue`가 담당합니다.

## 요청 예시

```text
$to-tickets 승인된 이슈 #50을 구현 가능한 작은 이슈들로 나눠 초안을 보여 줘.
```

실제 GitHub 이슈 생성까지 원한다면 “게시해 줘”라고 분명히 요청해야 합니다.
