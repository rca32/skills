# `writing-great-skills`: 예측 가능한 스킬 작성하기

정확한 실행 계약: [`skills/writing-great-skills/SKILL.md`](../../skills/writing-great-skills/SKILL.md)

## 한마디로

Codex 스킬을 새로 설계하거나 기존 스킬을 다듬어 **호출되는 상황, 실행 절차와 완료 판단이 매번 예측 가능하도록 만드는 작성 가이드**입니다.

좋은 스킬은 문장이 길거나 친절한 스킬이 아니라, 표현이 조금 달라져도 에이전트가 같은 근거와 안전 경계를 따라 일하게 만드는 스킬입니다.

## 먼저 여섯 가지를 정합니다

- **Purpose:** 이 스킬이 소유하는 결과 하나
- **Trigger:** 어떤 요청에서 선택되어야 하는지
- **Inputs:** 시작에 필요한 맥락과 자료
- **Branches:** 상황에 따라 달라지는 주요 경로
- **Completion:** 실제 완료를 증명하는 관찰 가능한 조건
- **Non-ownership:** 이웃 스킬이나 바깥 흐름이 맡아야 하는 일

짧은 문단 안에서 이 여섯 가지를 설명하기 어렵다면 하나의 스킬에 너무 많은 책임이 들어갔을 가능성이 큽니다.

## 자동 호출과 명시 호출

자주 필요하고 호출 조건이 분명한 스킬은 모델이 요청을 보고 자동으로 선택할 수 있습니다. 반대로 외부 시스템을 변경하거나, 드물고 전문적이거나, 잘못 호출했을 때 비용이 큰 흐름은 사용자가 `$skill-name`으로 직접 부르는 명시 호출이 적합합니다.

frontmatter의 description에는 실제 행동, 구체적인 호출 상황과 이웃 스킬과 겹치지 않는 부정 범위를 담습니다. 호출 정책은 `agents/openai.yaml`에서 관리합니다.

## 정보는 필요한 높이에 둡니다

모든 내용을 `SKILL.md`에 넣지 않고 다음 단계 중 가장 적절한 곳에 둡니다.

1. 매번 실행하는 순서는 `SKILL.md`의 단계
2. 대부분의 경로에서 쓰는 규칙은 `SKILL.md`의 참고 규칙
3. 특정 분기에서만 필요한 상세 내용은 직접 링크한 `references/`
4. 반복적이고 실패하기 쉬운 기계 작업은 테스트 가능한 `scripts/`
5. 사용자 결과물에 복사할 재료는 `assets/`

핵심 흐름을 reference에 숨기지는 않습니다. 반대로 한 분기에서만 쓰는 상세 규칙 때문에 모든 호출의 context를 무겁게 만들지도 않습니다.

## 예측 가능한 단계의 특징

- 각 상태 전환에 시작 조건, 수행 작업, readback, 종료 조건이 있음
- 읽기, 로컬 수정, 외부 쓰기, 게시, 삭제 권한을 서로 구분함
- 외부 쓰기 결과가 불명확할 때 재시도보다 먼저 reconciliation을 정의함
- 파일 경로나 현재 구현 세부보다 공개 계약과 관찰 가능한 결과를 사용함
- 안전 금지 규칙에는 대신 사용할 안전한 경로를 함께 제시함

중복, 너무 넓은 책임, 겹치는 trigger, 행동을 바꾸지 않는 장식 문장은 제거합니다. 각 규칙은 가능한 한 하나의 권위 있는 위치에만 둡니다.

## 게시 전 검증

정답을 노출하지 않는 현실적인 prompt로 정상 경로뿐 아니라 다음 상황을 함께 시험합니다.

- 이웃 스킬이 선택되어야 하는 요청
- 모호하거나 없는 변경 권한
- 부분적으로 실패하거나 결과를 알 수 없는 외부 쓰기
- 같은 상태를 건드리는 동시 실행자
- 변경 중이거나 dirty한 작업 트리

그 후 구조 검사, 포함된 스크립트 테스트, Standards와 Spec을 분리해 리뷰합니다. 독립 실행 환경이 없으면 한 컨텍스트에서 두 기준을 순서대로 검토하고 그 사실을 밝히면 됩니다.

새 스킬을 실제 패키지 구조로 초기화하고 metadata를 검증하는 일은 `skill-creator`와 함께 진행합니다.

## 요청 예시

```text
$writing-great-skills 이 SKILL.md의 호출 조건이 겹치는지 확인하고 완료 기준을 더 명확하게 다듬어 줘.
```
